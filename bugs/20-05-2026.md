# Intervention Bugs - 20-05-2026

## 1. Overlay app name is always generic

Status: DONE BUT NOT TESTED

`WatchdogService` computes `activePackage`, but never assigns it to `currentForegroundPackage`. `InterventionOverlay.show(...)` receives `currentForegroundPackage`, so the overlay falls back to generic text like `STOP USING APP` instead of the actual app name.

Fix:

`WatchdogService.checkUsage()` now assigns the detected `activePackage` to `currentForegroundPackage` before escalation can trigger the overlay.

Testing instructions:

1. Install a debug build on Android.
2. Select a known app such as YouTube or Instagram for tracking.
3. Set the collective daily limit low enough to trigger quickly.
4. Open the tracked app and wait for an intervention.
5. Confirm the overlay button says `STOP USING <APP NAME>` instead of `STOP USING APP`.

## 2. Default global daily limit may never reach native service

Status: DONE BUT NOT TESTED

The React state defaults `globalDailyLimit` to `30`, but native shared preferences default to `0`. `WatchdogService` exits early when `globalLimitMinutes <= 0`, so interventions may never trigger unless the user changes the dashboard limit and persists it.

Fix:

`AppPreferencesModule` and `WatchdogService` now use the same native fallback of `30` minutes when `globalDailyLimit` has not been saved yet.

Testing instructions:

1. Clear app data or install fresh.
2. Complete onboarding without changing the dashboard daily limit.
3. Confirm the dashboard shows `30m`.
4. Let tracked app usage reach the limit, or temporarily lower the native/default limit in a debug build to speed this up.
5. Confirm `WatchdogService` no longer exits as if the limit were `0`, and interventions can trigger.

## 3. Escalation level is consumed before playback is confirmed

Status: DONE BUT NOT TESTED

`handleEscalation` updates `currentEscalationLevel` before `triggerVoiceNote` proves that a valid recording exists, the overlay can be shown, and audio playback can start. If playback fails, that level may be skipped for the rest of the session.

Fix:

`triggerVoiceNote(...)` and `VoiceNotePlayer.play(...)` now return `Boolean`. `handleEscalation(...)` advances `currentEscalationLevel`, updates `hasHitDailyLimitInitially`, and writes `lastInterventionTimeMs` only when a voice note was found and audio playback actually started.

Testing instructions:

1. Record a note for level 1 only; leave levels 2 and 3 empty.
2. Trigger level 1 and confirm it plays.
3. Continue usage until level 2 would trigger.
4. Confirm missing level 2 audio does not permanently consume level 2.
5. Add a level 2 recording, re-enter or continue the tracked app, and confirm level 2 can still trigger.

## 4. Overlay permission failure can block audio playback

Status: DONE BUT NOT TESTED

`triggerVoiceNote` calls `interventionOverlay.show(...)` and then `voiceNotePlayer.play(...)` inside one `try` block. If overlay display throws, audio playback is skipped too.

Fix:

Overlay display is now wrapped in its own `try/catch`. If overlay display fails, the error is logged, but `voiceNotePlayer.play(...)` still runs.

Testing instructions:

1. Complete onboarding and record at least one level 1 voice note.
2. Start the watchdog by reaching the dashboard with usage access enabled.
3. Revoke display-over-other-apps permission from Android settings.
4. Use a tracked app until intervention should trigger.
5. Confirm audio still plays even though the overlay cannot appear.

## 5. No explicit intervention cooldown

Status: DONE BUT NOT TESTED

`lastInterventionTimeMs` exists but is unused. Escalation checks prevent repeated playback at the same level, but there is no explicit cooldown guard if state resets or detection flickers.

Fix:

`WatchdogService` now enforces a 60-second intervention cooldown before triggering a new voice note. Successful interventions update `lastInterventionTimeMs`.

Testing instructions:

1. Set the daily limit low and use a tracked app until level 1 triggers.
2. Immediately leave and re-enter the tracked app, or switch rapidly between tracked apps.
3. Confirm another intervention does not trigger within 60 seconds.
4. Wait more than 60 seconds and continue usage until the next escalation threshold.
5. Confirm the next eligible escalation can still trigger.

## 6. Foreground app detection is fragile

Status: DONE BUT NOT TESTED

`getActiveForegroundPackage()` queries usage stats for the last minute and picks the highest `lastTimeUsed`. This can misidentify the active app compared with usage events.

Fix:

`getActiveForegroundPackage()` now reads recent `UsageEvents` and tracks the latest `MOVE_TO_FOREGROUND` / matching `MOVE_TO_BACKGROUND` state instead of guessing from aggregate usage stats.

Testing instructions:

1. Enable usage access and select two tracked apps.
2. Move between the tracked apps, the launcher, and an untracked app.
3. Watch `Watchdog` logs or intervention behavior.
4. Confirm sessions start only when the currently foreground app is tracked.
5. Confirm leaving the tracked app hides the overlay and stops audio on the next poll.

## 7. Watchdog does not restart after reboot

Status: DONE BUT NOT TESTED

The service starts from the app/dashboard flow only. There is no boot receiver, so interventions do not automatically resume after device reboot until the app starts the watchdog again.

Fix:

Added `BootReceiver` and `RECEIVE_BOOT_COMPLETED`. On boot, it restarts `WatchdogService` only when onboarding is complete and both usage access and overlay permission are available.

Testing instructions:

1. Install the app on Android and complete onboarding.
2. Grant usage access and display-over-other-apps permission.
3. Reboot the device or emulator.
4. After unlock, check that the watchdog foreground notification appears without opening the app.
5. Open a tracked app and confirm interventions can still trigger.
